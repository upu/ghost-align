# Release a Version

Take the accumulated `[Unreleased]` CHANGELOG entries through to a release, following this repo's GitHub Flow. `main` is protected: changes land only via a PR with the `test` CI check green. This skill is the counterpart to `/ship` (which adds entries) and follows the rules in `docs/changelog-guide.md` (Keep a Changelog 1.1.0 + SemVer).

`$ARGUMENTS` is the target version `x.y.z` (no leading `v`). If empty, propose the next version from the current `package.json` `version` and the nature of the `[Unreleased]` entries, and confirm with the user.

## Responsibility boundary

This skill goes **up to merging the release PR and confirming the Release workflow it auto-triggers succeeded**. Merging to `main` with a bumped `package.json` version is itself the trigger — there is no manual tag push. Publishing is delegated to `.github/workflows/release.yml` (#256), which fires on every push to `main`, and does nothing (no-op success) unless the pushed `package.json` version has no matching GitHub Release yet: it verifies the `VSCE_PAT`, packages the VSIX, **creates a GitHub Release with the `.vsix` attached** (this is what creates the `vx.y.z` tag — there's no separate tag-creation step), **publishes to the VS Code Marketplace when the `VSCE_PAT` secret is present** (otherwise that one step is skipped, not failed), and **closes the matching `vx.y.z` milestone once all its issues are closed** (#168 — otherwise it leaves the milestone open and warns, without failing the release). In the unlikely case the workflow is absent, fall back to a manual publish (`npm run package` then `npx @vscode/vsce publish`, which needs `VSCE_PAT` / `vsce login`) and say so in the report.

## Steps

1. **Pre-checks** — confirm the working tree is clean and `origin/main` is up to date (`git fetch origin`). Read the `## [Unreleased]` sections of `CHANGELOG.md` (English, canonical) and `CHANGELOG.ja.md`: if they have no entries, stop and confirm with the user (nothing to release); if only one file has entries, fix the desync first (`prepare-release` would reject it anyway). Restate the entries so the version scope is explicit.
2. **Confirm the version is SemVer-correct** — given the `[Unreleased]` entries, check `x.y.z` matches Semantic Versioning: breaking changes → major, new features (`Added`) → minor, only fixes → patch. If the requested number disagrees with the entries, flag it and confirm before proceeding. This judgment stays here, not in the script below — only the mechanical rewrite is scripted.
3. **Branch from latest main** — `git checkout -b release/v<x.y.z> origin/main`.
4. **Prepare the release** — run `npm run prepare-release -- x.y.z`. This deterministically finalizes both `CHANGELOG.md` and `CHANGELOG.ja.md` (renames `## [Unreleased]` to `## [x.y.z] - YYYY-MM-DD` using today's local date, reinserts a fresh empty `## [Unreleased]` above it, and updates the link references at the bottom: `[Unreleased]` repoints to `.../compare/vx.y.z...HEAD` and a new `[x.y.z]: .../releases/tag/vx.y.z` is added) and bumps `package.json`'s `"version"` — see #257/#269. It exits non-zero without writing any file if the version isn't valid SemVer, isn't greater than the current version, or either file's `[Unreleased]` has no entries (a one-sided entry is a desync to fix in the CHANGELOGs, not here); read the error and fix the input rather than hand-editing the files.
   - (Only if yanking a release: after running the script, manually mark the heading `## [x.y.z] - YYYY-MM-DD [YANKED]` in both files — the script has no dedicated support for this rare case.)
5. **Test (gate)** — run `npm run compile`, `npm test`, then `npm run check:package`. All must pass; do not open a PR on a failing build.
6. **Commit** — confirm only intended changes are staged (`git status` / `git diff`), then commit with a Japanese summary line in the repo's style, e.g. `release: vx.y.z` (or `release: CHANGELOG を x.y.z に確定し version を bump`).
7. **Push & PR** — `git push -u origin release/v<x.y.z>`, then `gh pr create --base main`. Pass the body via `--body-file` (avoid backticks/`$()` in the inline command so the call matches the `gh pr *` allowlist) summarizing the release.
8. **Wait for CI** — run `set -o pipefail; gh pr checks <pr> --watch --fail-fast 2>&1 | tail -5` as a single foreground call with a long timeout (e.g. 600000 ms / 10 min); it blocks until all checks finish. The `tail -5` trims `--watch`'s 10-second reprints of the full check table (a context flood over many PRs); `set -o pipefail` keeps a failing check exiting non-zero through the pipe. Keep it one self-contained call rather than a background fire-and-forget resumed across turns — that multi-turn pattern is where a tool-call syntax slip has stalled the session at CI-wait. Fix and re-push if anything is red.
9. **Wait for Copilot review, if one was requested** — this repo's branch ruleset can auto-request a Copilot review on every push (`copilot_code_review` rule, `review_on_push: true`), but that setting could be off at any time, so detect it per PR rather than assuming: `gh pr view <pr> --json reviewRequests,reviews --jq '{pending:[.reviewRequests[].login], done:[.reviews[].author.login]}'`. If `Copilot` appears in neither list, wait ~10s and check once more (the auto-request can lag slightly behind the push); if it's still absent, Copilot review isn't configured/applicable for this PR — skip straight to Merge. If `Copilot` is in `pending`, poll the same command with an until-loop (not a chained retry) every ~20–30s, up to ~10 min, until it moves out of `pending` and into `done` — Copilot reviews normally land within a couple of minutes.
    - Once it lands, read it: the overview (`gh pr view <pr> --json reviews`) and any inline findings (`gh api repos/:owner/:repo/pulls/<pr>/comments --jq '.[] | select(.user.login=="Copilot") | {path,line,body}'`). Fix genuine, actionable findings with another commit on the same branch and push again — this re-triggers CI and, per `review_on_push`, a fresh Copilot review, so repeat step 8 and this step once more. Reply to a finding instead of changing code when it's a false positive or out of scope, explaining why.
    - Only move on to Merge once nothing is outstanding — this is what prevents discovering Copilot's comments only after the release PR is already merged.
10. **Merge** — `gh pr merge <pr> --squash --delete-branch`, then `git checkout main && git pull` to sync local main. This merge — carrying the bumped `package.json` version — is itself what triggers `.github/workflows/release.yml` on the push to `main` (#256); there is no separate tag-push step.
11. **Verify the release** — confirm the auto-triggered run actually succeeded rather than assuming. Get it with `gh run list --workflow=release.yml -L 1` (the merge in the previous step is the most recent push to `main`, so this is the run to watch), then watch it with a single foreground `set -o pipefail; gh run watch <run-id> --exit-status 2>&1 | tail -20` call (long timeout, e.g. 600000 ms) — one self-contained call, not a background watch resumed across turns; `tail` trims the repeated job-progress reprints, and `--exit-status` + `pipefail` preserve a red run's non-zero exit. A green run means the GitHub Release (its `vx.y.z` tag created as a side effect) with the `.vsix` was created and, with `VSCE_PAT` set, the Marketplace publish ran.
    - If it is red, inspect with `gh run view <run-id> --log-failed`. Recovery depends on **where** it failed:
      - **Before the Release was created** (compile / package / `verify-pat`): fix the cause — via the normal GitHub Flow if it's a code issue, or nothing if it's transient — then `gh run rerun <run-id> --failed`. The workflow's own version-based gate makes this safe: nothing was created yet, so a full retry produces no duplicates.
      - **After the Release was created** (only the Marketplace publish step failed — the realistic case, since `verify-pat` already ran beforehand): a plain Re-run will *skip* the whole job (the gate sees the Release already exists) instead of retrying publish. Recover by publishing manually (`npm run package && npx @vscode/vsce publish --packagePath dist/ghost-align.vsix -p <PAT>`) or by `gh release delete vX.Y.Z --cleanup-tag --yes` (safe — Marketplace never got the publish) followed by `gh run rerun <run-id> --failed`. See `[[feedback_retag_when_unpublished]]` memory for the fuller rationale.
12. **Confirm the milestone closed** — `release.yml`'s "Close milestone" step (#168) closes the `vx.y.z` milestone automatically once the release succeeds, provided all its issues are closed; if issues remain open it leaves the milestone open and posts an `::warning::` in the run log instead of failing the release. Check the run log (from step 11) or `gh api repos/:owner/:repo/milestones --jq '.[] | select(.title=="vx.y.z") | .state'` to confirm the outcome. If it's still open because of leftover issues, report them so the user can decide (defer to the next version or pull them into this release) — do not close it manually to paper over open issues.
13. **Report** — state the merged PR number, the new `main` commit, the Release workflow result (link the run), whether the Marketplace publish ran or was skipped, and whether the milestone was closed.

## Notes

- `main` is protected (no direct push/force-push/delete); always go through a PR with the green `test` check.
- One release per PR. Do not bundle unrelated changes into the release PR — it should contain only the CHANGELOG finalization and the version bump (via `npm run prepare-release`).
- The release commit itself is not a user-facing change, so it gets no new `[Unreleased]` entry — it *creates* the released section.
- The `vx.y.z` tag is created by `gh release create` inside the workflow, not pushed manually; never force-update a tag that's already reflected in a published GitHub Release — if a release is wrong after Marketplace publish succeeded, cut a new patch version instead.
- The CI-wait, merge, and release-verify steps are where tool-call syntax slips have repeatedly stalled the session. Don't rely on noticing a non-dispatch after the fact — that reminder was tried and didn't hold. Instead structure those steps so they can't half-fire: one self-contained foreground call each (the blocking `gh pr checks --watch`, the single `gh pr merge`, the single `gh run watch`) — never a background fire-and-forget you depend on resuming across turns.
- The same ruleset's `copilot_code_review` rule does not review draft PRs (`review_draft_pull_requests: false`) — keep the release PR non-draft (the default from `gh pr create`) if step 9's wait is meant to apply.
